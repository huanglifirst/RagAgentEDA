# RagAgentEDA 项目说明

这是一份给零基础读者看的说明文档。目标不是解释每一行代码，而是让你先建立一个正确的整体认识：这个项目是什么、能做什么、现在做到哪一步、你应该从哪里开始看。

## 一句话定义

`RagAgentEDA` 是一个面向 EDA 场景的后端 Demo。它会先从 `Resource/` 里的 TED 文档中检索相关知识，再把这些知识喂给大模型，生成测试脚本，最后可选地执行脚本并返回结构化结果。

## 三个最重要的结论

1. 它要解决的问题是：把“看文档写测试脚本”这件事，变成“系统先检索文档，再自动生成测试脚本”。
2. 它现在已经能做到：读取本地文档、切块、向量化、二阶段重排、生成测试脚本，并通过后端接口返回证据与日志。
3. 它现在还做不到：所有任务类型都稳定接入真实 TED 执行模板；当前 `execute=true` 仅 `opamp_bandwidth` 走真实模板链路。

---

## 项目是做什么的

先用一句白话说清楚。

假设你是电路工程师，现在想让系统“测试一个运放的带宽”。传统做法是：

1. 自己去翻 TED 文档。
2. 找到相关 API 或操作步骤。
3. 手工写测试脚本。
4. 运行脚本，拿回指标。

这个项目想做的事，是把前两步尽量自动化：先从文档检索证据，再基于证据生成脚本。

所以它的核心目标不是“做聊天机器人”，而是“做一个会先查资料、再生成测试脚本的 EDA 后端原型”。

---

## 一个具体例子：用户输入“测试运放带宽”之后发生了什么

用户发起请求：

```json
{
  "query": "请测试该运放电路的带宽",
  "circuit_description": "两级运放，输出带负载",
  "top_k": 6,
  "execute": false
}
```

系统内部会做这几件事：

1. 在 `Resource/` 目录里找到可读文档。
2. 做文本抽取和切块（含标题感知切块）。
3. 根据文档 fingerprint 加载向量索引；在 `reindex` 场景会强制重建向量。
4. `stage-1` 向量召回：先找语义最接近的候选块。
5. `stage-2` 模型重排：用 `qwen3-rerank` 对候选重排。
6. 若 rerank 调用失败，回退 lexical rerank，并把 warning 写到 `logs.stderr`。
7. 挑出最相关证据，交给大模型生成脚本。
8. `execute=false` 时直接返回脚本与证据，不执行脚本。
9. `execute=true` 时会进入 `preflight -> execute`，执行前先 `source ~/.bashrc` 并做环境检查，再读取 `result.json`。

## 项目整体流程

下面把主链路压缩成 8 步，方便你建立心智模型。

1. 读取配置  
   从 `.env` 里读取路径、模型、API Base、API Key、执行模式等参数。

2. 扫描知识库  
   从 `Resource/` 里读取 `md`、`markdown`、`html`、`htm`、`txt` 文档。

3. 文本清洗与切块  
   HTML 做正文抽取与模板噪声过滤；按标题层级分段后再做滑窗切块（`size=1200`、`overlap=200`、`<50` 丢弃）。

4. 文档向量化  
   调用 embedding 接口把文本块转成向量（带 batch/retry）。

5. 向量索引持久化  
   保存到 `workdir/vector_index/<fingerprint>.json`，并更新 `LATEST`。

6. 检索与重排  
   `stage-1` 向量召回，`stage-2` `qwen3-rerank` 重排；失败时回退 lexical rerank。

7. 生成测试脚本  
   把检索证据交给大模型，让模型输出单个 Python 脚本。

8. 可选执行脚本  
   `execute=true` 时先做 `preflight`，再执行并校验 `result.json`（含 `metrics.bandwidth_hz`）。

## 核心目录说明

这里不做文件清单堆砌，只解释你最该知道的目录职责。

### `Resource/`

知识库原料目录。项目不会把原文直接喂给模型，而是先抽取、分段、切块后再检索。

### `backend/app.py`

后端服务入口，暴露：

- `/health`
- `/v1/rag/reindex`
- `/v1/tasks/run`

### `backend/config.py`

全局配置中心。会从 `.env` 读取并覆盖环境变量，后续模块统一使用 `settings`。

### `backend/rag/`

检索核心模块，负责：

- 文档扫描和切块
- embedding 与索引读写
- 向量检索与回退检索

### `backend/agents/`

任务编排和代码生成：

- `orchestrator.py`：`prepare -> retrieve -> codegen -> (preflight) -> (execute)`
- `codegen.py`：按检索证据生成脚本

### `backend/llm/`

OpenAI 兼容客户端封装，当前用到：

- `/chat/completions`
- `/embeddings`
- `/rerank`
- `/models`

### `backend/runner/`

脚本执行器（本地或 SSH），负责执行、抓取日志、校验 `result.json`。

### `workdir/`

运行中间产物目录，包含向量索引、任务脚本、`result.json`、stdout/stderr 日志等。

## 两个核心接口

### 1. `/v1/rag/reindex`

作用：重建 `Resource/` 的向量索引。

调用示例：

```bash
curl -X POST "http://127.0.0.1:8000/v1/rag/reindex"
```

当前真实行为（重要）：

1. 强制重建：`force_rebuild=True`，不是“仅首次向量化”。
2. 严格校验：必须满足 `vector_count > 0` 且 `vector_count == chunk_count`，否则接口返回失败（500）。
3. 索引文件包含元信息：`embedding_model`、`embedding_api_base`、`created_at`。

简化成功响应：

```json
{
  "ok": true,
  "doc_count": 300,
  "fingerprint": "abc123...",
  "chunk_count": 5200,
  "vector_count": 5200,
  "saved_dir": "./workdir/vector_index",
  "saved_file": "./workdir/vector_index/abc123....json"
}
```

### 2. `/v1/tasks/run`

作用：提交任务，做检索与代码生成，可选执行。

调用示例：

```bash
curl -X POST "http://127.0.0.1:8000/v1/tasks/run" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "请测试该运放电路的带宽",
    "circuit_description": "两级运放，输出带负载",
    "top_k": 6,
    "execute": false
  }'
```

关键参数：

- `query`：自然语言需求
- `circuit_description`：电路描述
- `top_k`：证据条数
- `execute`：是否执行脚本

`execute=false` 与 `execute=true`：

- `execute=false`：只检索+生成，不执行，便于调试检索与脚本质量。
- `execute=true`：走执行链路，但当前仅 `opamp_bandwidth` 接入真实模板；其它类型会明确报“未接入真实模板”。

## 向量化是怎么做的

相关文件：

- [backend/rag/indexer.py](D:/EDA/RagAgentEDA/backend/rag/indexer.py)
- [backend/rag/vector_store.py](D:/EDA/RagAgentEDA/backend/rag/vector_store.py)
- [backend/llm/client.py](D:/EDA/RagAgentEDA/backend/llm/client.py)

过程概要：

1. 扫描 `Resource/` 文档。
2. 文本抽取与分段（HTML 正文抽取 + 噪声过滤 + 标题感知）。
3. 滑窗切块（`1200/200`），过滤短片段。
4. embedding 批量调用并重试。
5. 保存 `chunks + vectors + 元信息` 到 JSON。

补充说明：

- 当前仍是 JSON 向量索引方案（非独立向量数据库）。
- `fingerprint` 由文件路径/大小/mtime 计算，用于索引一致性。
- `LATEST` 记录当前默认索引文件名。

## 代码生成是怎么做的

相关文件：

- [backend/agents/orchestrator.py](D:/EDA/RagAgentEDA/backend/agents/orchestrator.py)
- [backend/agents/codegen.py](D:/EDA/RagAgentEDA/backend/agents/codegen.py)

机制：

1. 检索返回 `evidence`（source/score/snippet）。
2. 代码生成器基于 `query + circuit_description + evidence` 生成脚本。
3. `execute=false` 返回预览脚本。
4. `execute=true` 时：
   - `opamp_bandwidth`：走真实模板，依赖 `TED_BANDWIDTH_CMD`。
   - 其他类型：当前未接入真实模板，返回错误而不是伪执行。

## 检索是怎么做的

相关文件：

- [backend/agents/orchestrator.py](D:/EDA/RagAgentEDA/backend/agents/orchestrator.py)
- [backend/rag/vector_store.py](D:/EDA/RagAgentEDA/backend/rag/vector_store.py)
- [backend/rag/retriever.py](D:/EDA/RagAgentEDA/backend/rag/retriever.py)

当前检索链路：

1. `stage-1` 向量召回（embedding）。
2. `stage-2` `qwen3-rerank` 重排。
3. 若 rerank 异常：回退 lexical rerank。
4. 若 embedding 异常：直接 lexical fallback。

降级提示位置：

- 明确写入响应 `logs.stderr`。
- 常见标记：`embedding retrieval unavailable`、`using lexical fallback`。

## 执行模式

相关文件：

- [backend/runner/local_runner.py](D:/EDA/RagAgentEDA/backend/runner/local_runner.py)
- [backend/runner/ssh_runner.py](D:/EDA/RagAgentEDA/backend/runner/ssh_runner.py)

### `local` 模式

- 先 `bash -lc 'source ~/.bashrc && ...'`。
- 执行 `python main.py`，采集 `stdout.log`、`stderr.log`。
- 校验 `result.json` 结构。

### `ssh` 模式

- 通过 SSH/SCP 上传脚本并远端执行。
- 执行前同样会 source 远端 bashrc（默认 `~/.bashrc`）。
- 拉回 `result.json` 与日志并做校验。

## 如何启动和调试

### 第一步：安装依赖

```bash
pip install -r requirements.txt
```

### 第二步：确认 `.env`（当前口径）

```env
# chat
OPENAI_API_BASE=https://ark.cn-beijing.volces.com/api/v3
OPENAI_API_KEY=...
MODEL_NAME=deepseek-v3-1-terminus

# embedding
EMBEDDING_API_BASE=https://a.fe8.cn/v1
EMBEDDING_API_KEY=...
EMBEDDING_MODEL_TEXT=text-embedding-v4

# rerank
RERANK_ENABLED=true
RERANK_MODEL_TEXT=qwen3-rerank
RERANK_API_BASE=https://a.fe8.cn/v1
RERANK_API_KEY=...
RERANK_TOPN_FACTOR=4

# runtime
RAG_RESOURCE_DIR=Resource
RAG_WORK_DIR=./workdir
RAG_VECTOR_INDEX_DIR=./workdir/vector_index
RAG_EXECUTION_MODE=local
RAG_LOCAL_BASHRC=~/.bashrc
RAG_REMOTE_BASHRC=~/.bashrc
```

说明：`EMBEDDING_MODEL_TEXT` 默认值是 `text-embedding-v4`；最终以你本地 `.env` 实际值为准。

### 第三步：启动后端

```bash
uvicorn backend.app:app --host 0.0.0.0 --port 8000
```

### 第四步：重建向量索引

```bash
curl -X POST "http://127.0.0.1:8000/v1/rag/reindex"
```

重点检查：

- `chunk_count > 0`
- `vector_count > 0`
- `vector_count == chunk_count`

### 第五步：先跑 `execute=false`

```bash
curl -X POST "http://127.0.0.1:8000/v1/tasks/run" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "请测试该运放电路的带宽",
    "circuit_description": "两级运放，输出带负载",
    "top_k": 6,
    "execute": false
  }'
```

重点看：

- `generated_code` 是否非空
- `evidence` 是否相关
- `logs.stderr` 是否包含降级 warning

### 第六步：再跑 `execute=true`

仅在你确认检索与脚本质量可接受后再执行。注意真实执行能力当前只覆盖 `opamp_bandwidth`。

## 怎么判断项目是否正常

### 1. 服务层

```bash
curl "http://127.0.0.1:8000/health"
```

建议确认字段：

- `ok`
- `model/model_api_base/model_api_key_set`
- `embedding_model/embedding_api_base/embedding_api_key_set`
- `rerank_enabled/rerank_model/rerank_api_base/rerank_api_key_set`
- `vector_index_latest`

### 2. 索引层

调用 `/v1/rag/reindex` 后确认：

- `doc_count > 0`
- `chunk_count > 0`
- `vector_count == chunk_count`
- 返回非 500

### 3. 检索与生成层

调用 `/v1/tasks/run` 且 `execute=false` 后确认：

- `generated_code` 非空
- `evidence` 有内容
- `logs.stderr` 无降级提示（理想情况）

### 4. 执行层

`execute=true` 时确认：

- `result.json` 存在且结构有效
- `metrics.bandwidth_hz` 为数值（`PASS` 时应 `>0`）
- `stdout/stderr` 可定位问题

## 当前局限与下一步

### 当前局限

1. 当前是后端 Demo，主要用于流程验证。
2. 向量索引是 JSON 持久化，不是独立向量数据库。
3. 真实执行模板目前只覆盖 `opamp_bandwidth`。
4. 运行效果依赖模型可用性与 Key 权限。
5. 执行器依赖 bash/bashrc 与目标环境依赖完整性。

### 下一步建议

1. 先稳定 embedding/rerank/chat 的模型可用性。
2. 用 `execute=false` 持续做检索回归与证据质量评估。
3. 扩展真实模板到 SFDR、环路增益等任务类型。
4. 再考虑更重的工程化能力（如更强存储/治理/观测）。

## 总结

如果只记住一句话，请记住这句：

**这个项目不是完整 EDA 平台，而是“先检索 TED 文档，再生成测试脚本，并可选执行”的后端原型。**

它已经打通了主链路，适合做能力验证与迭代；但在“多任务类型真实执行模板”上还处于逐步落地阶段。
