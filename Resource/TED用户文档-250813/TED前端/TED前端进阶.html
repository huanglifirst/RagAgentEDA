TED 前端进阶

本篇文档在前序“TED 前端基础”之上，进一步深入介绍 TED 模块构建的内部机制、节点和实例的高级用法，以及更复杂的连接模式，旨在帮助用户更高效、灵活地运用 TED 进行电路设计。

一、TED 模块是怎样构建的

使用 @module 装饰器定义模块时，TED 并非立即执行被装饰的函数 Func 来构建模块实体。实际上，@module 会将 Func 转换并包裹成一个模块构建函数 ModuleWrapperFunc。考虑到用户定义的 Func 函数体执行可能涉及复杂的逻辑和大量的实例创建，其执行时间代价可能较高，TED 设计了一套包含缓存机制的构建流程，以提高效率。

1. 模块构建流程与缓存

当调用模块（如 MyModule(...)）时，TED 会执行以下构建流程：

核心原则： 对于具有相同模块来源（路径）、相同模块名和相同参数值的模块调用，TED 只会执行一次完整的构建过程。后续对同一模块（相同 name_hash）的调用将直接复用已构建好的模块定义进行实例化，跳过耗时的构建步骤。

构建阶段:

  1. 预构建 (Pre-build):

    • 解析 Func 函数的签名（函数头），区分端口（无默认值参数）和参数（有默认值参数），并记录其类型和维度。
    • 根据模块路径（唯一标识模块源文件位置的信息）、模块名 (Func.__name__) 和调用时传入的所有参数值，计算一个唯一的哈希标识符 name_hash。这个 name_hash 将贯穿整个流程，作为该特定模块定义的唯一身份牌。重要提示： 为确保缓存的可靠性和 name_hash 的一致性，建议模块参数（带默认值的函数入参）使用 Python 中可哈希 (hashable) 的基本类型，如 int, float, str, bool, tuple 等。使用复杂或不可哈希类型作为参数可能导致未定义的缓存行为。
    • 以计算出的 name_hash 为键，在全局模块库 (Module Library) 中查询是否存在已构建的同名模块。
    • 缓存命中： 如果找到匹配的 name_hash，说明该模块定义已存在。直接使用库中缓存的模块定义来创建新的实例，并跳过后续的 Build 和 Post-build 阶段。
    • 缓存未命中： 如果未找到匹配项，则表示这是一个新的模块定义（或参数组合），需要进行完整构建：
      • 创建一个新的模块对象 (Module Object)。
      • 将这个新模块对象压入当前的模块调用栈 (Module Stack)。在当前模块出栈之前，可以通过 current_module() 函数访问到它。
      • 根据函数签名中解析出的端口定义，为当前模块创建相应的端口节点。

  2. 构建 (Build):

    • 执行用户代码: 执行用户定义的 Func 函数体。
    • 自动归属: 在 Func 函数执行期间创建的所有节点 (Node(...)) 和实例 (例如 Inv(), Nand2(), 或其他子模块实例) 都会被自动添加到 current_module() 所代表的当前模块对象中。
    • 可以通过 current_module().instances() 访问当前模块内的所有实例列表。
    • 可以通过 current_module().all_nodes() 访问当前模块内的所有节点集合。

  3. 后构建 (Post-build):

    • 自动重命名: 自动执行 current_module().auto_rename() 操作（详见后文“节点的名字”部分）。此操作会根据节点和实例在 Func 代码中的变量名，为那些尚未被手动 rename() 的节点和实例进行命名。注意： 此步骤默认执行，即使在 Func 函数体内已经调用过 auto_rename()，它依然会执行，但不会覆盖已经通过 rename() 或先前 auto_rename() 获得的名称。
    • 计算当前模块的尺寸（通常与版图信息相关，细节待补充）。
    • 构建当前模块所依赖的子模块列表。
    • 注册模块: 使用在 Pre-build 阶段计算出的、唯一的 name_hash 作为键，将完整构建好的当前模块对象注册到全局模块库中，以便后续调用时能够缓存命中。
    • 将当前模块对象从模块调用栈中弹出，完成本次模块构建过程。

示例演示:

import logging
# (假设 debug 是配置好的 logging 函数)
from ted import module, Input, Output, Ctrl, Node, current_module, get_circuit, testbench

# 注：示例中使用的 get_circuit() 是 TED 框架提供的全局函数，
# 用于获取代表当前整个电路设计的 Circuit 对象，
# 通过它可以访问电路的全局状态，例如 module_stack。
def test6():
    @module
    def Func(a: Input, b: Output, c: Ctrl, w=1e-6, h=1e-6):
        # Build 阶段执行
        logging.debug(
            f"构建模块：{current_module().name_hash()} w = {w}, h = {h}"
        )
        logging.debug(f"模块调用栈：{'() '.join(map(str, get_circuit().module_stack))}()")

    @module
    def Top():
        # 调用 Func, 第一次触发 Func(w=1e-6, h=5e-6) 的构建
        Func(w=1e-6, h=5e-6)
        # 再次调用，参数相同，缓存命中，不触发 Func 构建
        Func(w=1e-6, h=5e-6)
        # 调用 Func，参数不同，触发 Func(w=1e-6, h=2e-6) 的构建
        Func(w=1e-6, h=2e-6)

        logging.debug(f"模块 {current_module()} 的实例：")
        for inst in current_module().instances():
            logging.debug(inst)

    # 实例化 Top 模块，触发其构建流程
    Top()
    # Top 构建完成后，打印最终的模块栈 (通常只剩 ROOT)
    logging.debug(f"最终模块调用栈：{'() '.join(map(str, get_circuit().module_stack))}()")

(预期输出分析保持不变)

二、节点描述进阶

1. 节点的名字

TED 中的每个节点 (CircuitNode) 都有一个名字。

  • 自动命名: 如果不指定，TED 会自动为节点生成一个唯一的名字，通常格式为 n<数字>_ (例如 n2_)。末尾的下划线用于区分系统自动名和用户指定名。
  • 手动命名: 可以使用节点的 rename() 方法给节点指定一个名字：
    a = Node()
a.rename("my_signal")
  • 自动变量名命名 (auto_rename): 通常，我们希望节点的名字能直接反映其在 Python 代码中的变量名。可以通过 current_module().auto_rename() 实现这一点。该方法会遍历当前模块内所有通过变量赋值创建的、且尚未被手动 rename() 的节点和实例，并将它们的 Python 变量名赋给它们。
    def test1():
    @testbench
    def Top():
        input_a = Node()
        output_z = Node()
        logging.debug(f"手动 auto_rename 前：{input_a}, {output_z}") # 可能输出 n2_, n3_
        # 手动调用 auto_rename (例如，需要在中间步骤获取确定性名称)
        current_module().auto_rename()
        logging.debug(f"手动 auto_rename 后：{input_a}, {output_z}") # 输出 input_a, output_z

        # 即使手动调用过，Post-build 阶段仍会执行 auto_rename，
        # 但不会覆盖已有的 'input_a', 'output_z' 名称。
    重要: 即使用户在模块函数体中调用了 auto_rename()，在模块构建的 Post-build 阶段，系统仍然会执行一次 auto_rename()。这个默认行为确保了所有符合条件的节点和实例最终都能获得基于变量名的名称，但它不会覆盖已经被 rename() 手动指定或先前 auto_rename() 赋予的名称。在函数体中调用 auto_rename() 的主要场景是，如果需要在后续代码中依赖这些基于变量名的、可预测的节点/实例名称。

2. 复杂节点 (Bundles)

CircuitNode 不仅可以表示单个电学节点，还可以表示节点的集合，即“复杂节点”或“总线”(Bundle)。

  • 定义多节点:

    # 创建一个包含 3 个独立普通节点的复杂节点
bus_a = Node(3) # 元素索引为 0, 1, 2
x, y, z = bus_a # 将元素解包到独立变量

# 定义索引范围和步长
data_bus = Node(7, 0) # 长度为 8，索引为 7, 6, ..., 1, 0 (逆序)
control_sig = Node(5, 8) # 长度为 4，索引为 5, 6, 7, 8
even_bits = Node(0, 6, 2) # 长度为 4, 索引为 0, 2, 4, 6
    • Node(n): 创建长度为 n 的复杂节点，索引 0 到 n-1。
    • Node(start, stop): 创建长度为 abs(stop - start) + 1 的复杂节点，索引从 start 到 stop（支持正序和逆序）。当 start == stop 时 (如 Node(5, 5))，创建一个长度为 1 的复杂节点，其唯一元素索引为 start (例如可以通过 n[5] 访问)。
    • Node(start, stop, step): 在 start 到 stop 范围内按 step 步长创建索引。其精确长度（包含的普通节点数）通常取决于起点、终点和步长。例如，对于 step > 0 且 stop >= start 的常见情况，长度可通过 floor((stop - start) / step) + 1 计算得出（其他情况如逆序或负步长需相应调整）。

  • 检查长度: len(bus_a) 返回复杂节点包含的普通节点数量。

  • 构建嵌套复杂节点: 可以将已有的普通节点或复杂节点组合成新的复杂节点。

    a = Node(3)
b = Node(2)
c = Node()
# m 是一个包含 a(复杂节点), b(复杂节点), c(普通节点) 的新复杂节点
m = CircuitNode.new_bundle_from([a, b, c])
logging.debug(f"Length of m: {len(m)}") # 输出 3 (m 本身包含3个元素: a, b, c)

  • 展平 (Flatten): 将复杂节点（无论多少层嵌套）转换成一个只包含普通节点的一维列表。

    n = m.flatten()
logging.debug(f"Length of flattened m: {len(n)}") # 输出 6 (3 + 2 + 1)

  • 复杂节点命名:

    • 手动 rename() 依然有效，会为整个 Bundle 设置基础名称，但其内部元素通常需要通过索引访问。
    • auto_rename() 会为 Bundle 变量（如 bus_a）命名，并自动为其元素命名，格式通常为 var_name<index> (e.g., bus_a<0>, bus_a<1>, bus_a<2>)。
    • 未命名的复杂节点，系统会自动命名，其元素名可能类似 n2_<0>_, n2_<1>_。

3. 节点连接规则

  • 简单节点连接 (a >> b):

    • 连接后，a 和 b 指向同一个底层节点对象。
    • 命名优先级:
      1. 若 a, b 均未被 rename -> 合并后节点使用 a 的原始自动名。
      2. 若 a, b 中只有一个被 rename -> 使用那个被 rename 的名字。
      3. 若 a, b 都被 rename -> 使用 b 的名字 (这是一个设计约定)。
      4. 若连接中涉及字符串 (e.g., a >> "my_net" 或 "my_net" >> a) -> 字符串被视为一个已命名的节点参与上述规则。

  • 复杂节点连接 (bundle_a >> bundle_b):

    1. 对 bundle_a 和 bundle_b 执行内部展平 (flatten)。
    2. 比较展平后两个列表的长度。
    3. 长度不匹配: 抛出 DimensionMismatchError 异常 (或类似异常)，指示连接错误。用户有责任确保连接双方长度一致。
    4. 长度匹配: 按顺序将 bundle_a 展平后的第 i 个元素与 bundle_b 展平后的第 i 个元素进行简单节点连接。

  • 节点与元组/列表的连接 (bundle_a >> (x, y, z)):

    • TED 允许将节点（简单或复杂）连接到 Python 的元组 (tuple) 或列表 (list)。
    • 自动处理嵌套结构: TED 会递归地遍历右侧的元组/列表结构，收集所有节点元素，并按照线性展开的顺序构建一个临时的扁平化复杂节点。
    • 然后，按照复杂节点连接规则将左侧节点与这个临时构建的复杂节点进行连接（包括长度检查和逐元素连接）。
    complex_node = Node(6)
# 假设 x,y 是 Node(), z 是 Node(2), w 是 Node(2)
# ((x, y), z, w) 会被自动展平成包含 6 个节点的临时 Bundle
complex_node >> ((x, y), z, w) # 连接成功 (如果 x,y,z,w 定义如上)

  • 实例连接 (node >> instance, instance >> node, instance_a >> instance_b):

    • 遵循通用规则：A >> B 表示将 A 的输出连接到 B 的输入。
    • 节点：其输入和输出都是自身。
    • 实例：其输入和输出由其模块定义中的 Input 和 Output 端口决定（可能为复杂节点）。
    • 连接时，系统会提取 A 的输出端口节点（或 A 自身如果是节点）和 B 的输入端口节点（或 B 自身如果是节点），然后应用简单或复杂节点连接规则。

4. Serial 运算符

Serial 提供了一种更清晰地描述长串联路径（纯数据流）的方式，作为 >> 链式写法的替代。

# 等效于 a >> b >> c >> d
Serial(a, b, c, d)

# 复杂例子
Serial(
    Vdd(),
    PMOS(...) % ctrl_p, # 注意：Serial 不处理 % 连接，这里的 % 是 PMOS 自身的构造
    intermediate_node,
    NMOS(...) % ctrl_n,
    Vss(),
)

重要: Serial 仅用于描述数据流向 (等效 >>)，不能用于描述控制信号连接 (不等效 %)。

三、实例描述进阶

1. 实例列表 (InstList)

InstList 是一个包含多个有序实例的容器对象，常用于表示阵列、重复结构等。

  • 特性:

    • 可以像单个实例一样进行操作（移动 move_to, 翻转 flipx, 旋转 rotate, 布局 row/col/lay, 设置电源 set_vdd/set_vss, 导出 to_gds 等）。这些操作会同时应用于列表中的所有实例。
    • 支持高级功能，如 smartbus（智能总线布线，细节待补充）。
    • 可以通过索引访问内部的单个实例。

  • 创建方式:

    • 直接构造: inst_list = InstList([inst1, inst2, ...])
    • 实例复制 (* 运算符):
      # 创建一个 Inv 实例作为模板
inv_template = Inv()
# 创建一个包含 5 个新的、独立的 Inv 实例的 InstList
inv_array = inv_template * 5
# 重要：执行 * k 操作时，原始的模板实例 (inv_template) 会被自动销毁 (kill)。
# 这个自动 kill() 操作不仅销毁实例对象，还会解除该模板实例所有端口
# 与电路中其他节点的任何现有连接，将其从设计中完全移除。
# 因此，强烈建议用于 * k 复制的实例应作为纯粹的一次性模板；
# 如果复用了已连接的实例作为模板，用户必须清楚该实例及其连接将被彻底移除的后果。
assert isinstance(inv_array, InstList)
assert len(inv_array) == 5
# 此时 inv_template 变量可能仍然存在，但不代表一个有效的 TED 实例

2. 销毁实例 (kill)

在某些情况下，需要手动移除一个已经创建的实例，可以使用 kill() 方法。其效果如上所述，会断开连接并从设计中移除实例。

temp_inst = MyModule()
# ... 获取尺寸或其他信息 ...
temp_inst.kill() # 从设计中移除该实例及其连接

3. 批量连接与节点复制

  • 节点引用复制 (node * k):

    common_gnd = Node().rename("gnd")
# 创建一个包含 5 个元素的复杂节点，所有元素都指向同一个 common_gnd 节点
gnd_bus = common_gnd * 5
assert len(gnd_bus) == 5
# 连接 gnd_bus[0] 实际上就是连接 common_gnd
# gnd_bus[0] is gnd_bus[1] is common_gnd # 结果为 True

    注意: node * k 创建的是引用的集合。所有元素共享同一个底层节点。这与 Node(k)（创建 k 个独立节点）和 inst * k（创建 k 个独立实例）的行为截然不同，使用时需特别注意。

  • 批量连接实例: 可以将复杂节点/元组与 InstList 进行连接，实现并行数据通路的简洁描述。

    num_bits = 8
inputs = Node(num_bits)
outputs = Node(num_bits)
inv_array = Inv() * num_bits # 创建包含 8 个独立 Inv 实例的 InstList

# 将 inputs<0..7> 分别连接到 8 个 Inv 实例的输入
# 将 8 个 Inv 实例的输出分别连接到 outputs<0..7>
inputs >> inv_array >> outputs

    连接时，会按顺序将 inputs 的元素连接到 inv_array 中实例的输入，并将 inv_array 中实例的输出连接到 outputs 的元素。长度必须匹配。

  • DEC4TO16 示例解读: 这个复杂的解码器例子充分利用了嵌套元组的自动展平、节点/实例复制和批量连接特性，用高度向量化的方式描述了门级连接。

四、模块描述进阶

1. 定义多位端口

模块的端口也可以是多位的（复杂节点），定义方式与 Node() 类似，在端口类型声明中加入长度或范围参数即可。

@module
def MyAdder(A: Input(4), B: Input(4), CIN: Input, SUM: Output(4), COUT: Output):
    # A, B, SUM 都是长度为 4 的端口 (索引 0 到 3)
    # CIN, COUT 是普通单比特端口
    pass

@module
def ReversedBusModule(DIN: Input(7, 0), DOUT: Output(7, 0)):
    # DIN, DOUT 是长度为 8 的逆序端口 (索引 7 到 0)
    pass

端口类型 (Input, Output, Ctrl) 都支持这种方式定义多位端口。

2. 定义端口数可变的模块 (柯里化)

有时模块的端口数量需要根据参数动态确定。可以利用 Python 的函数柯里化 (Currying) 特性实现：定义一个外层函数接收配置参数（如位宽 N），内部再使用 @module 定义实际的模块，其端口宽度依赖于外层函数的参数。外层函数返回这个内部定义的模块。

def ParameterizedAdder(N=8):
    """返回一个 N 位加法器的模块定义"""
    @module
    def _Adder(A: Input(N), B: Input(N), CIN: Input, SUM: Output(N), COUT: Output):
        # ... N 位加法器的内部实现 ...
        pass
    return _Adder

# 实例化一个 16 位的加法器
# ParameterizedAdder(16) 返回模块定义
# () 执行该定义进行实例化
adder_16bit = ParameterizedAdder(16)()

实例化方式: 需要两对括号。第一对 (N=...) 调用外层函数并传入参数，获得模块定义；第二对 () 对返回的模块定义进行实例化。

3. 处理多电源域

当一个模块需要处理多个电源域（如 AVDD/AVSS, DVDD/DVSS）时：

  • 将这些额外的电源网络定义为普通的模块端口 (Input 或 Output 类型，根据电流方向）。
    @module
def MixedSignalBlock(
    analog_in: Input, digital_in: Input(4),
    analog_out: Output, digital_out: Output(8),
    AVDD: Input, AVSS: Input, # 模拟电源端口
    DVDD: Input, DVSS: Input  # 数字电源端口
):
    # ... 内部实现 ...
    analog_instance = AnalogPart(...)
    digital_instance = DigitalPart(...)

    # 使用 set_vdd/set_vss 将内部实例连接到正确的电源端口
    analog_instance.set_vdd(AVDD)
    analog_instance.set_vss(AVSS)

    digital_instance.set_vdd(DVDD)
    digital_instance.set_vss(DVSS)
  • 在模块内部，对于需要连接到特定电源域的子实例，使用实例的 set_vdd() 和 set_vss() 方法将其默认的 VDD/VSS（体端连接）重定向到对应的电源端口节点（如上例中的 AVDD, AVSS, DVDD, DVSS）。
  • 模块本身仍然具有隐藏的、默认的 VDD/VSS 端口（通常连接到其父模块的 VDD/VSS），如果需要，仍然可以使用 Vdd() 和 Vss() 访问。增加的电源端口本质上只是普通的信号端口，只是用途特殊。不存在“改变模块默认 VDD/VSS”的操作，而是通过端口和实例重定向来管理多电源。
